## 1. エージェントのセキュリティとガードレール

### OpenClawのセキュリティが向かう先
**原題**: Where OpenClaw Security Is Heading
**URL**: https://openclaw.ai/blog/where-openclaw-security-is-heading

ファイル操作やコマンド実行の権限を持つAIエージェントを安全に動かすための、OpenClawの多層防御ロードマップ。パストラバーサルを防ぐファイルシステム境界（fs-safe）、SSRFを防ぐネットワーク出口管理（Proxyline）、ClawHubによるプラグイン信頼評価、Tree-sitterでシェルコマンドの構造を解析する承認最適化、そして140超のOpenGrepルールとCodeQLによる静的解析の5本柱で構成される。特に、難読化された破壊的コマンドを検知しつつ「プロンプト疲れ」による安易な許可を抑える設計思想は、エージェント運用を考える全チームに示唆を与える。実装レベルで「権限の強いAIをどう囲うか」を具体化した好例だ。

### アイデンティティ・プロトコルはエージェント向けに設計されていない
**原題**: Open AI's Fotis Chantzis on why identity protocols weren't designed for agents
**URL**: https://1password.com/blog/agent-identity-protocols

OAuthやOIDCは「ユーザーの意図が安定している」前提の静的な仕組みだが、エージェントは実行中にタスクを自己拡張し、当初想定しないリソースへアクセスする「意図のドリフト」を起こす。OpenAIのFotis Chantzis氏は、標準化を待たずに今すぐ導入すべき4つの制御を挙げる——アクションごとに権限を評価する継続的認可、複数システムを跨いでも元の人間まで遡れる帰属追跡、モデルに秘密を直接渡さない資格情報の仲介、そしてタスクに特化した短寿命・限定スコープの認証だ。エージェントを本番投入する前に読むべき、認可設計の現実的な指針。

### CloudflareでClaude Managed Agentsが利用可能に
**原題**: Announcing Claude Managed Agents on Cloudflare
**URL**: https://blog.cloudflare.com/claude-managed-agents/

CloudflareとAnthropicの提携により、AIの「脳」をAnthropic側に、コード実行やネットワーク接続という「手」をCloudflareのサンドボックスに分離して運用できるようになった。V8 Isolateベースのミリ秒起動・低コスト構成と、フルLinuxのMicroVM構成を選べる。カスタムプロキシによる認証情報の注入とデータ漏洩防止、Workers VPC経由での社内サービスへの安全な接続、ブラウザ操作やメールツールの標準搭載が揃う。「知能」と「実行環境」を分離して安全にスケールさせるという、エージェント実行基盤の現実解を示した発表だ。

### LLMCap：API利用額に「物理的な上限」をかける
**原題**: LLMCap — Hard Dollar Caps on LLM API Calls
**カテゴリー**: ツール・実験
**URL**: https://www.llmcap.io/

予期せぬ高額請求を防ぐコスト管理プロキシ。最大の特徴は、予算上限に達したとき「通知」ではなく、プロバイダーへ転送する前にHTTP 429で「遮断」する点にある。SDKのbase_urlを一行書き換えるだけで導入でき、追加レイテンシは35ms未満。APIキーをサーバーに保存せずリクエストごとにパススルーする設計で、Anthropic・OpenAI・Gemini・Mistral・Cohereに対応する。エージェントの暴走的なトークン消費が現実のリスクとなった今、「ソフトな警告」では足りないという割り切りが効いている。

### Claw Patrol：エージェント向けOSSセキュリティ・ファイアウォール
**原題**: Claw Patrol: an open-source security firewall for agents
**カテゴリー**: セキュリティ・リスク
**URL**: https://deno.com/blog/clawpatrol

Deno製のセキュリティ・ファイアウォール。既存のLLMゲートウェイがHTTP通信のみを対象とするのに対し、Claw PatrolはWireGuard/Tailscaleでネットワーク層に位置し、SQLやKubernetesといった非HTTPプロトコルまで解釈・制限できる。エージェントに実際の認証情報を持たせない「クレデンシャル・インジェクション」、HCLによるルール設定、Human-in-the-loopの承認フローをサポートし、DROP TABLEやリソース削除といった破壊的操作を未然に防ぐ。前述のPocketOS事件のような事故を構造的に防ぐ、ネットワーク層からのアプローチとして注目に値する（現在MITライセンスのアルファ版）。

### Slow Mode：あえて「遅い」エージェントの提案
**原題**: Slow Mode
**URL**: https://blog.val.town/slow-mode

Val TownのPete Millspaugh氏が、自動化・高速化を追う「YOLO Mode」がもたらすコード理解不足とスロップの蓄積に対し、人間をプロセスの中心に置く「Slow Mode」を提唱する。AIが勝手にループを回すのではなく、ステップごとに計画を共有し、関数の命名や構造の選択を人間に問い、概念を教えながら進める。生産性と引き換えに「学習」と「長期的なメンテナンス性」を得るという発想だ。料理でレシピを機械的にこなすのでなく素材を学びながら創作するように、一生モノのスキルを育てる——速度礼賛への静かな対抗軸として読みたい。

### ワークフローの衝突：人間の運用とエージェントのライフサイクル
**原題**: The Workflow Collision
**URL**: https://webframp.com/posts/the-workflow-collision/

カンバン方式に代表される人間のワークフローは信頼と自律を前提とした「プル型」で最小限のステートで回る。対してエージェントのライフサイクルは、起動・詳細な事前計画・多段階の検証ゲートを要する「制約型」だ。両者を一つのボードに無理に統合すると、人間の柔軟性が失われるかエージェントの安全装置が壊れる。著者は、マージではなく、人間のワークフローの中にエージェントのライフサイクルをサブプロセスとして内包させる「合成（コンポジション）」を推す。人間が「何を作るか」を統治し、エージェントが「どう作るか」を厳格に実行する分業の設計図だ。

### AIを5本同時に走らせても、脳みそは1個
**カテゴリー**: 教育・学習
**URL**: https://zenn.dev/pepabo/articles/9baa6271b2055d

Claude Code等で5〜6本の並列開発が可能になったが、人間の脳は本来並列処理ができない。筆者は認知科学の視点から「注意残余」と集中力の喪失を警告する。ワーキングメモリは実質3〜4チャンクであり、AIの高速出力に追従しようとすると報酬系が即時フィードバックに最適化され深い思考が犠牲になる。対策として、セッションを3〜4に制限し、同一メンタルモデルで扱えるタスクをまとめ、割り込みを事前に潰し、「監視すべき判断」と「放置できるタスク」を区分する。AIが加速するほど人間の判断力の価値が上がる——その原資たる集中力を守れ、という実務的な警句。

## 4. AIへの懐疑 — 限界と批判

### LLMの「越えられない壁」が数学的に証明された
**カテゴリー**: 批判的分析
**URL**: https://www.gizmodo.jp/2026/05/limits-of-ai-capabilities-oe8.html

Vishal Sikka氏らの論文が、LLMには特定の複雑さを超えるタスクを計算上の制約から完遂できない「上限」が数学的に存在すると示した。AI企業の謳う「無限の可能性」や、複雑な手順を自律的に進める「エージェント型AI」の実現性に冷水を浴びせる内容だ。Appleの研究者も「LLMは推論しているのではなく、推論しているように見せているだけ」との見解を示しており、本研究はそれを数学的に裏づける。AGIへの直線的進化という楽観論への、理論からのカウンターとして押さえておきたい。

### AI時代のコンピューター技術書
**カテゴリー**: 批判的分析
**URL**: https://golden-lucky.hatenablog.com/entry/2026/05/17/170605

LLMの真のインパクトは「生成」より、未整理の文章からも文意を汲む高度な「読解」にある——という鋭い着眼から出発する論考。訓練された読み手であるLLMの登場で、人間向けの「編集」を介さずとも情報消費が可能になった。だが人間が真に理解するには依然として丁寧な編集が要る。問題は経済だ。LLM要約が一般化する未来では、自ら読む人間のためだけに編集された本はコストが変わらないのに市場が縮小し、出版社は厳しい合理性に直面する。AI時代の技術書は従来通り編集を必要としながら、その存立基盤が揺らぐという逆説を突く。

### ロボットのための公教要理
**原題**: A Catechism for Robots

### 4つのAIにラジオ局を運営させてみた
**原題**: We let four AIs run radio stations. Here's what happened.
**カテゴリー**: ツール・実験
