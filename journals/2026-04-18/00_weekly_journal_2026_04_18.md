## 今週のハイライト

4月第3週は、Anthropicが Claude Opus 4.7・Routines・Desktop 再設計の3連続リリースを行い、Cloudflare が「Agents Week」として10本以上のエージェント向けインフラ記事を一斉公開した。開発ツールの大型アップデートという賑やかさの裏側では、まだ一般公開されていないモデル「Claude Mythos Preview」の危険性が英国 AI 安全研究所（AISI）の評価報告として明らかになり、米財務省・FRB が金融機関トップを集めてリスク警告を行うという異例の事態が起きた週でもある。

日本語コミュニティでは「ハーネスエンジニアリング」という概念の定義をめぐる議論が活発に展開された。「モデルの賢さよりも、エージェントを動かす環境設計が勝敗を決める」という主張が複数の独立した記事で展開されており、概念の共通言語が形成されつつあることが観察できる。Cloudflare や Microsoft、Vercel がそれぞれエージェントインフラの標準を打ち出したこととあわせて読むと、「エージェントを走らせる環境」が今年の技術的主戦場であることが浮かび上がる。

一方で、AI コーディングツールの普及による「人的コスト」の告発も目立った。シニアエンジニアの認知過負荷とバーンアウト、AI 生成コードの「理解の負債」、ヴァイブコーディングの医療現場での失敗事例など、生産性向上の裏側にある構造的な問題が具体的なデータとともに報告された。オープンソース側では、Alibaba の Qwen3.6-35B-A3B が Simon Willison のベンチマークで Claude Opus 4.7 の SVG 生成精度を上回り、ローカル実行可能な中規模モデルがフロンティアモデルに肉薄し始めたことが示された。

「賢いモデルを作る」から「そのモデルが最大限に機能する環境を設計する」へと開発の重心が移ってきた今週の動向は、AI 開発が成熟期に入りつつあることを告げている。同時に、能力向上がそのまま安全保障上のリスクと直結するという現実が、Mythos の評価報告という形で改めて可視化された。便利さと危険性の距離が縮まり続けるなか、私たちは何をどう設計するかを問われている。

## 1. Claude Opus 4.7・Routines・Desktop 再設計 — Anthropicの4月大型アップデート

Anthropicは今週、フラッグシップモデル Claude Opus 4.7 のリリースに加え、CLI ツール Claude Code に「Routines」機能とデスクトップ版の UI 再設計を相次いでリリースした。Routines はスケジュール実行・API トリガー・GitHub Webhooks の3形式をサポートし、人間の承認なしに繰り返しタスクを処理する。サブエージェントの公式ガイドと Opus 4.7 を Claude Code で使うベストプラクティスも公開され、エコシステム全体が新モデルを中心に刷新された。

### Claude Opus 4.7の発表：コーディング、視覚、自律性を大幅に強化したAnthropicの最新モデル

https://www.anthropic.com/news/claude-opus-4-7

Anthropic が公式リリースした「Claude Opus 4.7」は、特にソフトウェアエンジニアリングと自律的なタスク遂行において Opus 4.6 から大きな進化を遂げている。今回の発表で最も実務的なインパクトが大きいのは、コーディングベンチマークでの 13% のスコア向上だろう。単なる数値の改善ではなく、複数ファイルにまたがるリファクタリングや曖昧な仕様のバグ修正など、従来は人間が介入しなければならなかったタスクで自律性が増したことを意味する。

画像処理面では、長辺 2,576 ピクセルまでの高解像度対応により、スクリーンショット内の細部テキストやデザインモックアップのレビューがより正確になる。UI 設計のフィードバックや障害対応時のログスクリーンショット解析など、視覚情報を扱うワークフローで恩恵を受ける場面が増えるはずだ。

実用上とくに注目したいのが「xhigh」エフォートレベルの導入と「タスク予算」機能だ。推論の深さをタスクの重要度に応じて制御できるようになった点は、コストと性能のトレードオフを開発者が自ら調整できることを意味する。xhigh はデフォルト設定であり、API 設計や大規模リファクタリングのような重いタスクに最適化されている。一方、タスク予算はトークン消費に上限を設けることでコスト管理を可能にし、大量のエージェントを並列実行する際の予算制御に直結する。

指示遵守の厳格化も重要な変更点だ。以前のモデルより回答が簡潔になり、曖昧な指示を独自に解釈して過剰な処理を行うことが減った。実務的には「意図しない副作用が起きにくくなった」という評価に繋がる一方、プロンプトが明示的でないと期待した処理を行わないケースも出てくる。指示の粒度を上げることが改めて重要になる。

セキュリティ面では、Cyber Verification Program の開始が注目される。正当なセキュリティ研究者に対して、従来は拒否されていたような脆弱性研究の支援を行う仕組みで、モデルの能力向上と責任ある利用のバランスを取る試みだ。価格は Opus 4.6 から据え置きだが、新しいトークナイザーにより入力トークン消費量が 1.0〜1.35 倍程度変動する可能性があり、既存のコスト見積もりの見直しが必要になる。

### Claude Code デスクトップアプリが刷新、複数エージェントの並行実行に対応

https://claude.com/blog/claude-code-desktop-redesign

新しいモデルを最大限に活かすには、それを動かす環境の刷新も必要だった。Anthropic は今週、Claude Code のデスクトップ版 UI を大幅にアップデートし、複数の AI エージェントを同時に走らせる現代的なワークフローへの最適化を図った。

最大の変更は、進行中のセッションを一覧管理できる新しいサイドバーの追加だ。複数のリポジトリで並行して作業する際に、それぞれのエージェントの進捗を一つの画面で把握できる。ドラッグ＆ドロップによるレイアウトカスタマイズ、アプリ内で直接実行できる統合ターミナルとエディタ、メインの会話を邪魔せずにコンテキストを参照できる「サイドチャット」機能も追加された。大規模な差分を高速に処理するリビルドされた diff ビューア、Mac と Linux 両対応の SSH によるリモートマシン操作も実用面で効いてくる機能だ。これにより、ブラウザや外部エディタを行き来することなく、一つの画面で複数リポジトリの修正・テスト・デプロイを指揮できる環境が整った。

### Claude Codeに「ルーチン」機能が登場：GitHub連携や定期実行による開発自動化が可能に

https://claude.com/blog/introducing-routines-in-claude-code

並列実行の環境が整ったところに、今度は「人間がいなくても動く」仕組みが加わった。Claude Code の新機能「ルーチン（routines）」は、プロンプト・リポジトリ・コネクタを一度設定すれば、スケジュール実行・API・GitHub Webhooks 経由で自動稼働させられる機能だ。夜間のバグトリアージ、デプロイ後のスモークテスト、PR 作成時の自動レビューといった定型ワークフローが対象になる。

重要な特徴は、ルーチンが Claude Code のクラウド基盤上で動作するため、ユーザーの PC がオフラインでも実行される点だ。現在はリサーチプレビュー段階で Pro、Max、Team、Enterprise プランのユーザーに提供されており、プランごとに1日の実行回数制限がある。自律的な定期実行という性質上、安全策の設計が今後の鍵になるだろう。

### Claude Opus 4.7 を Claude Code で使いこなすためのベストプラクティス

https://claude.com/blog/best-practices-for-using-claude-opus-4-7-with-claude-code

## 2. Claude Mythos Preview と AI安全保障 — 40社限定公開に至る経緯とAISI評価

Anthropicの未公開モデル「Claude Mythos Preview」について、英国 AI 安全研究所（AISI）が公式評価を公開した。CTF 課題で 73% の成功率を記録し、32 段階の企業ネットワーク攻撃シミュレーションをフロンティアモデルとして初めて完遂した。米財務省・FRB が主要銀行 CEO に警告を行うなか、Anthropic は供給チェーンリスク指定に対する控訴も棄却されており、モデルの能力拡大と規制・安全保障の緊張が高まっている。

### Claude Mythos Previewのサイバー攻撃能力に関するAISIの評価報告

https://www.aisi.gov.uk/blog/our-evaluation-of-claude-mythos-previews-cyber-capabilities

英国 AI 安全研究所（AISI）が公開した Claude Mythos Preview の評価報告は、これまでのフロンティアモデルの能力評価の中で最も具体的な危険性を示したドキュメントの一つだ。報告の核心にある数字から見ていこう。エキスパートレベルの Capture-The-Flag（CTF）課題において 73% という成功率を記録した。CTF とはセキュリティ競技で、脆弱性の発見・悪用・フラグ奪取という一連の攻撃プロセスを模したものだ。この成功率は、専門家の熟練ハッカーと比較しても遜色のないレベルに達していることを意味する。

しかし最も重大なのは「The Last Ones（TLO）」と呼ばれる企業ネットワーク攻撃シミュレーションの完遂だ。32 のステップを要するこの複雑な攻撃シナリオを、フロンティアモデルとして初めて最初から最後まで完遂した（10 回中 3 回）。TLO は単一の脆弱性を突く単純な攻撃ではなく、偵察・侵入・横断移動・目標達成という現実の APT（高度持続的脅威）を模した多段階シナリオだ。これを自律的に完遂できるというのは、AI が実際のサイバー攻撃に使用可能な段階に達したことを示している。

もう一つ注目すべきは推論トークン予算のスケーリング効果だ。推論に使えるトークン数を増やすほど性能が向上し、1億トークンまでのスケールが確認されている。これは計算資源を増やすことで、より難しい攻撃課題も解決できるようになることを意味する。コスト低下と並行して攻撃能力も向上するという、セキュリティの観点から見て懸念すべき動向だ。

一方で報告は重要な限界も示している。評価は「能動的な防御が存在しない環境」で行われており、セキュリティ監視やインシデント対応が整った実際の組織環境での有効性は未知数だ。AISI はこの結果を踏まえ、AI が脆弱なシステムへの自律的な脅威となり得る段階に達したと警告し、パッチ適用やアクセス制御という「セキュリティの基本」の徹底を推奨している。

## 3. ハーネスエンジニアリング論争 — 「環境設計」が競争優位になる時代

「ハーネスエンジニアリング」の定義をめぐり、日本語コミュニティで複数の独立した記事が展開された週だった。watany 氏がハーネスという用語の混乱を整理し、sasadango28 氏が5層構造を提示し、murasaki 氏はエージェント開発の成熟度を8レベルで定義した。mkj 氏は「RL 環境の共有」まで含めた「大環境時代」の到来を論じ、Ubie の事例ではセキュリティエージェント向けのハーネス設計が実装レベルで報告されている。

### AIエージェントの「ハーネス」に関わる混乱と私見

https://zenn.dev/watany/articles/d8b692bbca65a3

「ハーネスエンジニアリング」という言葉を使う前に、その言葉が何を指しているのかを確認したほうがいい。これが watany 氏の記事の核心だ。AI エージェントの文脈で急速に普及している「ハーネス」という用語は、立場によって大きく意味がズレている。著者は Martin Fowler の記事を出発点に、その分類を「内部ハーネス」と「外部ハーネス」に整理した。

「内部ハーネス」はエージェントの作り手側の視点だ。LangChain の「Agent = Model + Harness」という定義が象徴するように、LLM 以外の実装（メモリ管理、ツールオーケストレーション、ループ制御など）を指す。LangChain はこの文脈でハーネスを「自社製品」として提供している。一方 Anthropic は、将来的にモデルが賢くなれば補助的な内部ハーネスは「枷」になると主張する。同じ「ハーネス」という言葉を使いながら、LangChain はそれを不可欠な基盤と見なし、Anthropic はいずれ不要になる補助輪と位置づけている。

「外部ハーネス」は利用者側の視点だ。Mitchell Hashimoto や OpenAI の文脈では、エージェントが同じミスを繰り返さないよう「環境側で恒久的な修正を施す」設計指針を指す。エージェントへのフィードバックループそのものがハーネスであり、プロンプトを修正するのではなく環境を設計するという発想だ。ベンダーが自社製品へのロックインを意図して「ハーネス」という言葉を使う場合と、実務的な安全利用を求めるユーザーが使う場合で意味が乖離している。自分がどちらのハーネスの話をしているかを明確にすることが、このテーマの入口として必要だ。

### Claude Codeでハーネスエンジニアリングを実践する — 5層の設計パターン

https://zenn.dev/sasadango28/articles/claude-code-harness-engineering-20260415

用語の地図ができたところで、Claude Code を舞台に5層の実装論を見てみよう。sasadango28 氏は Claude Code の流出ソースコード分析を踏まえ、ハーネスエンジニアリングを CLAUDE.md・Rules・Skills・Agents・Settings の5レイヤーで体系化した。各レイヤーの役割は明確に分かれている。CLAUDE.md はプロジェクトの目的と「やらないこと」を定義する憲法。Rules は glob パターンで条件付き読み込みされるドメインルール。Skills はカスタムスラッシュコマンドで、LLM を「判断」に・スクリプトを「決定的な処理」に分離することでトークン消費を抑える。Agents は特定役割に特化したサブエージェントで、メインコンテキストを汚染しない並行処理を担う。Settings はツール実行権限を制御する安全装置だ。この5層構造は、LLM を「信頼性の高い開発環境の一部」として組み込む設計思想を具体化している。

### 2026年4月時点のエージェンティック・エンジニアリングとハーネス設計の現在地

https://zenn.dev/murasaki/articles/d9fea8b4e730ce

## 4. エージェント・インフラの拡張 — CloudflareのAgents WeekとMicrosoft APM

Cloudflare が「Agents Week」と銘打ち、エージェント向けメモリ・ブラウザ・Email・AI 検索・Durable Object Facets など10本以上の機能を一斉公開した。同時期に Microsoft は「APM（Agent Package Manager）」というエージェントのツールと権限セットを npm のように管理するパッケージシステムを発表し、Vercel はサーバーレスの「Durable Execution」モデルを提案した。これらは「エージェントが実行環境・状態管理・ツール依存関係を持つ独立したソフトウェア単位として扱われる時代」への移行を示す。

### 「Agents Week」の幕開け：AIエージェント時代のインターネット・インフラを構築する

https://blog.cloudflare.com/welcome-to-agents-week/

Cloudflare が「Agents Week」で示したビジョンの核心は、インターネットのアーキテクチャそのものを問い直すことだ。従来のクラウドインフラは、一つのアプリケーションが多数のユーザーを支える「1対多」のモデルで設計されてきた。しかしエージェントは、1人1人に固有の実行環境と状態管理を必要とする「1対1」のモデルだ。ユーザー 100 万人がそれぞれの AI エージェントを持つ世界では、従来のコンテナ技術はスケーラビリティとコストの限界に直面する。

Cloudflare の答えは V8 Isolate を活用した Workers 技術だ。コンテナより 100 倍高速かつ効率的という主張は誇張ではなく、起動レイテンシと実行環境の軽量性において本質的な差がある。各エージェントが数ミリ秒で起動し、独立した環境で実行され、終了後に即座にリソースを解放できる。これがスケールする「エージェントのためのインフラ」の骨格だ。

発表された機能は多岐にわたる。エージェントが既存の Web を操作するためのブラウザレンダリング機能、次世代の標準となりつつある Model Context Protocol（MCP）のサポート、エージェント間の経済取引を可能にする HTTP 402 対応、そして Zero Trust セキュリティとの統合だ。MCP への投資は特に戦略的意味合いが大きい。エージェントが使えるツールとデータソースの標準インターフェースとして MCP が普及すれば、Cloudflare のインフラ上でそれらが動く価値が増大するからだ。

Agents Week 全体を通じて見えてくるのは、Cloudflare がエージェントのランタイム・通信・経済・セキュリティという4つのレイヤーすべてを自社で賄おうとしているという野心だ。単なる CDN やサーバーレス実行基盤という役割を超え、「エージェントが生きるインターネット」のインフラプロバイダーへの転換を宣言している。

### Dynamic WorkersにおけるDurable Objects：AI生成アプリごとに専用データベースを付与する

https://blog.cloudflare.com/durable-object-facets-dynamic-workers/

ビジョンを支える具体的な技術がある。Durable Object Facets は「エージェントが生成したコードに専用の SQLite DB を」という発想を実装したものだ。動的にロードされた Worker コード（AI が生成したアプリなど）を「Facet」として実行し、各インスタンスに独立したデータベースを自動付与する仕組みで、API を介したプロビジョニングなしに超低レイテンシなステートフルアプリをオンデマンドでデプロイできる。親となる Durable Object が監視・ロギング・課金を担い、子となる Facet がそれぞれ独立した状態を持つ設計は、AI が生成した動的コードを安全にマルチテナント実行する際の有力なパターンだ。

## 5. AIコーディングの人的コスト — バーンアウト・認知負債・「10倍」の実態

AI コーディングツールの普及が「10倍の生産性」をもたらす一方で、シニアエンジニアが AI 生成コードのレビュー負荷で物理的な燃え尽きに追い込まれているとの告発記事が話題となった。O'Reilly の Addy Osmani は「理解の負債（Comprehension Debt）」という概念を提示し、コード生成速度が人間の理解速度を超えることで生じるリスクを論じた。ヴァイブコーディングの医療現場での失敗事例や、AI 支援による認知発達への影響など、生産性の裏側にある構造的課題が複数の角度から報告された週だった。

### AIによる「10倍の生産性」がシニアエンジニアを物理的に破壊する理由

https://techtrenches.dev/p/the-human-cost-of-10x-how-ai-is-physically

「10倍の生産性」というフレーズが語られるとき、暗黙の前提は「エンジニア全体が10倍になる」だ。だがこの告発記事が描く現実はまったく異なる。AI はコードを生成し、プルリクエストの量を 98% 増加させた。しかし人間の脳の論理的処理速度は秒間約 10 ビットという生物学的な限界があり、そこは変わっていない。

この非対称性が「監督税（Supervision Tax）」という形でシニアエンジニアに集中的に課せられる。機械が数秒で生成したコードの妥当性を、シニアエンジニアが数時間かけて検証することを強いられる。しかも AI 生成コードは人間が書いたものよりバグが 1.7 倍多く、構造的な欠陥が 90% を占めるというデータがある。表面的には正しく見えるが深部に欠陥があるコードのレビューは、ゼロから書くよりも高い認知負荷を要求する。正確さを担保するために「本当に理解するまで追う」必要があるからだ。

問題はパフォーマンスの劣化だけにとどまらない。記事が挙げる心血管疾患リスクの増大、極度の疲労といった身体的コストは、長期的な人材損耗を引き起こす。「10倍の生産性」が実際には「シニアエンジニア1人あたりのレビュー負荷10倍」であり、その差分はシニアの疲弊によって吸収されているという構造だ。技術の問題ではなく、組織設計の問題として捉え直す必要があると記事は指摘する。

### AI「バイブ・コーディング」の恐怖：素人がAIで構築した患者管理システムの崩壊

https://www.tobru.ch/an-ai-vibe-coding-horror-story/

### 理解の負債：AI生成コードに潜む隠れたコスト

https://www.oreilly.com/radar/comprehension-debt-the-hidden-cost-of-ai-generated-code/

個別の失敗を超えた構造的問題がある。Google の Addy Osmani はそれを「理解の負債（Comprehension Debt）」と名付けた。システム内に存在するコード量と、人間が実際に理解している内容との間に生じるギャップだ。AI は人間が評価・理解できる速度よりも遥かに早くコードを生成できる。特に、ジュニア開発者がシニアの監査能力を超える速度でコードを量産する「速度の非対称性」が問題を深刻化させる。Osmani が提唱するのは、AI を受動的な代行者ではなく概念的な探究のパートナーとして活用し、システム全体のメンタルモデルを維持する「理解のプロセス」の重視だ。速く動かすことと、それを理解することは別の営みだというシンプルな事実を、これだけの生産性向上の時代に改めて確認させてくれる。

https://heidenstedt.org/posts/2026/ai-assisted-cognition-endangers-human-development/

負債が積み重なると、人間の思考そのものが変質する。著者は人間の知識の進化を「動的弁証法基質（Dynamic Dialectic Substrate）」で説明し、既存の概念が衝突・融合することで新しい概念が生まれるプロセスを定義する。しかし LLM は学習データの統計的パターンに固執する「帰納バイアス」を持ち、ユーザーが AI と繰り返し議論・壁打ちすることで、思考の範囲が AI のバイアス内に収束してしまう「AI スキュー」が発生すると指摘する。これが集団レベルで起きれば「認知のインブリーディング」だ。人間の創造的・適応的思考が阻害される危険性だ。対策として著者は人間同士の対話の重視と、AI の回答に誘導されない「認知衛生（Cognition Hygiene）」の維持を提唱する。個別ツールの利便性を超えた、使い方の哲学を問う議論として、このテーマを締めくくる。

## 6. ローカルLLMとオープンソース競争 — Qwen3.6がClaude Opus 4.7を超えた瞬間

Alibaba が4月16日に公開した Qwen3.6-35B-A3B は、35B パラメータ・3B アクティベーション（MoE）設計で SWE-bench Verified 73.4% を達成し、ラップトップで動作する。Simon Willison の SVG 生成ベンチマークでは、Claude Opus 4.7 が自転車フレームの描画に失敗した一方で Qwen3.6 が正確に描写し、特定タスクでの逆転が観測された。一方、Meta は音楽生成モデル「Muse Spark」をクローズドな API として公開し、Llama 路線からの転換を示した。

### Qwen3.6-35B-A3B：エージェントコーディング性能、オープンに解放

https://qwen.ai/blog?id=qwen3.6-35b-a3b

Alibaba の Qwen チームが4月16日に公開した Qwen3.6-35B-A3B は、ローカル実行可能なモデルとフロンティアモデルの境界線を塗り替える一作として記録されるかもしれない。「35B-A3B」という名称が示すのは、パラメータ総数 35B に対して推論時には約 3B のみをアクティブ化するスパース MoE（Mixture of Experts）設計だ。3B 相当の計算コストで 35B の表現力を引き出すこの設計により、M2/M3 MacBook Pro やミドルレンジの GPU 搭載 PC でも実用的な速度で動作する。

性能面では、コーディング特化指標 SWE-bench Verified で 73.4% を記録した。これは同じく注目されている Gemma 4-31B の 52.0% を大きく上回る数値だ。Terminal-Bench 2.0 で 51.5%、ツール使用を測る MCPMark で 37.0% というスコアも、エージェントタスクへの実用性を裏付ける。コンテキスト長はネイティブ 262K トークンで最大 100 万トークンまで拡張可能、テキスト・画像・文書・動画をアーキテクチャレベルでネイティブサポートするマルチモーダル設計も備える。

ライセンスは Apache 2.0 で商用利用が無料という点も見逃せない。プロプライエタリモデルへの API 依存を避けたい組織、プライバシー上の理由でローカル実行が必要な環境、またはコスト管理を重視するプロジェクトにとって、フロンティア水準のエージェントコーディング能力がローカルで手に入ることの意義は大きい。前世代（Qwen3.5-35B-A3B）を全ベンチマークで大幅に上回ったこのモデルは、「コモディティ化しつつあるフロンティア性能」という今の時代を象徴している。

Qwen だけではない。Gemma 4 もローカルでのエージェント CLI 動作が実証されている。Google Cloud のエンジニアが Gemma 4（26B MoE および 31B Dense）を Codex CLI で検証した結果、ツール呼び出し能力が 86.4% まで向上し、コード生成・テスト実行・パッチ適用というエージェント型ワークフローがローカルで完結できるようになったことが確認された。M4 Pro 搭載 MacBook Pro と NVIDIA Blackwell GPU 搭載 Dell GB10 の2環境で検証され、MoE アーキテクチャが Mac 環境で特に高速なトークン生成を実現することが示された。クラウドモデルの信頼性には及ばないものの、コスト・プライバシー・オフライン動作の観点でローカル実行が実用的な選択肢になったことが、複数モデルの実証で裏付けられている。

## おわりに

今週の6つのテーマを横断して見えてくるのは、AI 開発の「重力の移動」だ。モデルそのものの性能競争は今も続いているが、その上のレイヤー——エージェントを動かす環境設計、インフラの標準化、依存関係の管理——が急速に重要性を増している。ハーネスエンジニアリングの議論も、Cloudflare・Microsoft・Vercel のインフラ発表も、その同じ方向を向いている。「何を作るか」から「何がそれを動かすか」へと、エンジニアが問うべき問いのレイヤーが上がっている。

同時に、今週は AI の「裏側のコスト」が可視化された週でもあった。Mythos の評価報告が示すサイバーリスク、シニアエンジニアへの認知的・身体的負荷、Comprehension Debt の蓄積、そして医療現場でのヴァイブコーディングの失敗。これらは互いに独立した問題ではなく、「AI がより速く、より強力になるほど、それを扱う人間と社会への要求も増大する」という一つの構造を示している。

Qwen3.6 が Opus 4.7 の SVG 生成を上回り、Meta がオープンソース路線から離脱するという今週の動向は、AI の「オープンか、クローズドか」という問いが今後も揺れ続けることを告げている。ラップトップで動くモデルがフロンティアに迫る一方で、囲い込みの力学も働く。どちらの流れが主流になるかは、単なる技術の話ではなく、誰が AI を制御できるかという問いに接続している。
