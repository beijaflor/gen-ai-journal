一方で、AI生成速度に対する批判的論点も同じ強さで立ち上がった。和田卓人氏が「認知負債」を最大のリスクとして指摘し、O'Reilly、sshh.io、404 Mediaが揃って経営層の効率化主張と現場の保守負担の乖離を論じた。コード側では、Bryan Cantrillが「怠惰の美徳」の喪失を、James Shoreが保守コストの罠を、todeskingが「AI slop」長文化を、そしてrust-langが正式にLLM生成PR禁止ポリシーを示し、生成量の増加が招く長期的負債が複数角度から検証された。

セキュリティ側では、Anthropicの推論特化型モデル「Claude Mythos」の限定公開を巡る各国・各社の動きが目立った。安野貴博氏が政府対応の遅さを批判し、3メガバンクがアクセス権を確保した一方、NISTは中国製DeepSeek V4 Proを米国最先端から8ヶ月遅れと定量評価、22歳のKye Gomez氏は公開論文だけから「OpenMythos」を再現公開した。さらにGoogleがAI支援で発見したゼロデイの報告、MetabaseによるOSS脆弱性「露天掘り」時代の現場レポート、90日開示ポリシーの機能不全を訴える論考が、AI駆動スキャンが攻防の双方を加速させている実例として揃った週である。

## 1. Addy Osmani・dely・Codex・Google Skillsが具体化するハーネス／Agent Skills設計

AIエージェントの性能はモデル単体ではなく「ハーネス」と「Skills」の設計で決まるという議論が、今週は複数の実装事例として共有された。Addy OsmaniのO'Reilly寄稿による体系的整理、クラシルのRailsチームによる失敗のチームルール昇格、Codexサブエージェントによる実装・レビュー分業、Google公式Agent Skillsリポジトリ、そして子ども向けアプリ開発における「作らない判断」と、理論と現場実装の両面からハーネス／Skills時代の設計原則を確認できる週となった。

### エージェント・ハーネス・エンジニアリング：AIモデルを真のツールに変えるための設計手法

https://www.oreilly.com/radar/agent-harness-engineering/

GoogleのAddy Osmani氏がO'Reilly Radarに寄稿した本記事は、AIエージェント構築における中心概念を「Agent = Model + Harness」という等式で整理している。優れたモデルでも貧弱なハーネスでは機能せず、逆に適切なハーネスがあれば標準的なモデルでも高度なタスクを完遂できる。Osmani氏が指す「ハーネス」とは、システムプロンプト、ツール群、サンドボックス、可観測性、そしてオーケストレーションを含む周辺設計の総体である。

特に強調されているのが「スキル問題」としての再定義だ。エージェントの失敗をモデルの限界と決めつけるのではなく、コンフィギュレーション（設定）の欠落として捉え、失敗が発生するたびに`AGENTS.md`等のルールを更新し、二度と同じミスをさせない「ラチェット（歯止め）」の仕組みを組み込んでいく。これはエージェント運用を「個別のモデル選択問題」から「累積的な設計問題」へとフレーミングし直す視点である。

実装面では、コンテキスト窓の飽和を防ぐための要約（Compaction）、ツール実行結果の外部ファイル化、セッションのクリーンな再起動といった高度な制御が求められると述べる。さらに長期実行のループ設計として、計画（Planning）、自己検証（Self-verification）、生成と評価を分離するマルチエージェント構造（Generator/Evaluator）を挙げ、長期的なタスクの完遂率を高める方法論を提示している。

将来像としてOsmani氏が予測するのは「HaaS（Harness as a Service）」の登場である。開発者はLLM APIを直接叩くのではなく、ハーネスが提供するランタイムを活用する形態へとシフトしていく、というシナリオだ。ハーネスという語彙自体が業界共通の設計用語として定着しつつある現状を踏まえると、この記事は「ハーネス設計」という新たなエンジニアリング規律の出発点として位置づけられる内容である。読者にとっては、自分のエージェント運用がモデル依存になっているのか、ハーネス側で吸収できる構造になっているのかを点検する起点になる。

### エージェントを強化する: Google 公式 Skills リポジトリを発表

https://cloud.google.com/blog/ja/topics/developers-practitioners/level-up-your-agents-announcing-googles-official-skills-repository

この体系化を受けて、Googleは公式リポジトリでツール側からの足場提供に踏み込んだ。Google Cloud Next 2026で発表された「Agent Skills」の公式リポジトリ（github.com/google/skills）は、AIエージェントに特定の技術やタスクの専門知識を付与するためのMarkdown形式の凝縮されたドキュメント群である。

これまでMCP（Model Context Protocol）サーバーを利用する際に課題となっていた「コンテキストの肥大化」とそれに伴うコスト増・精度低下を解決するため、SkillはMCPの代替ではなく補完として位置づけられている。初期リリースではAlloyDB、BigQuery、GKE、Gemini APIといった主要プロダクトに加え、セキュリティやコスト最適化を扱うWell-Architectedの指針を含む13のSkillが公開された。開発者は`npx skills install`コマンドで自身のエージェントにこれらを組み込むことができる。Osmani氏の理論的整理と同じ週に公式ツールが揃ったことで、Skills設計は抽象論から具体実装の比較対象へと一段降りた。

Claude Code等のAIをフル活用しつつも、AIが得意とする「機能追加」の提案に対し、人間が「何を作らないか」を定義する重要性が強調されている。「予定」という言葉を「できごと」に抽象化して時制の矛盾を解消した例や、左右対称なモード構成をあえて崩す判断、メモリリーク調査におけるドメイン前提の共有など、AIとの協業における実戦的な知見が並ぶ。AIは提案を行い、人間は「責務」に基づいて決断を下す——という役割分担を、具体的なプロダクト開発の文脈で言語化した内容である。

### Claude Codeの失敗をチームルールに昇格させる仕組み

https://zenn.dev/dely_jp/articles/5bc3e9cf62d776

責務の切り分けを実運用に落とすには、失敗の蓄積と昇格の仕組みが必要になる。クラシルのRails開発チームによる本記事は、AIエージェント（Claude Code）が起こした失敗を蓄積・分析し、再発頻度の高いものを自動的にチーム共有の開発ルールへ昇格させる「ハーネスエンジニアリング」の実践手法を紹介している。

Mitchell Hashimotoが提唱する「エージェントのミスに合わせて環境を更新する」思想に基づき、3つのステップが構築されている。まずEvaluatorが検知した失敗をローカルの`FAILURES.md`に蓄積。次に`/failure-promote`コマンドにより、3件以上重なった失敗パターンをチーム共有の`.claude/rules`へ昇格させるPRを自動生成する。さらにセッション終了時に`/retro`を実行することで、PRレビューや手修正の履歴からカテゴリ別に知見を抽出し、Memory・Hook・Skillsへと還元する。Osmani氏の「ラチェット」概念を、実運用パイプラインとして具体化した事例である。

### Codexのサブエージェントで実装とレビューを分業してみた

https://acro-engineer.hatenablog.com/entry/2026/05/12/120000

ルールの自動昇格と並行して、エージェント分業による品質担保のパターンも報告されている。OpenAIのコーディングエージェント「Codex」におけるサブエージェント機能を用いた本記事は、実装担当とレビュー担当を別々のサブエージェントとして定義し、役割に応じて「思考の深さ（Reasoning Effort）」「MCP」「システムプロンプト」を個別に最適化する設定方法を解説している。

AngularとFastAPIを用いたアプリ開発の例では、シングルエージェントでは見落としがちな非同期処理の競合といった設計上の不備が、高精度設定のレビュー専任エージェントによって的確に指摘・修正されるプロセスが示された。さらに実験的機能「spawn_agents_on_csv」を利用することで、ジョブの投入から多段階の修正・再レビューの履歴をCSVで構造的に管理・可視化できる。

5本を並べて読むと、「ハーネス／Skills」が2026年中盤の設計用語として定着し、抽象論ではなく具体実装の比較対象となった現状が見えてくる。Osmaniの理論的フレームワークと公式ツール（Google Skills）、現場知見（dely・acro-engineer）、設計判断軸（mrksye）が同週に揃ったことで、業界標準化と現場最適化が同時進行している。読者にとっては「責務で引き算」「失敗の自動昇格」「分業」といったハーネス設計の具体原則を、自分のプロジェクトの判断軸として持ち帰れる週となった。

## 2. Agent View・Fast Mode・VS Code 1.120・Anthropic公式ガイド・cafleetが拡張するエージェント実行環境

複数エージェントの並列実行と監視を支える機能・知見が、ベンダー側とコミュニティ側の双方から同時期に揃った。Claude CodeのAgent View・Fast Mode、VS Code 1.120のAgentsウィンドウStable化、Anthropic公式の大規模コードベース運用ガイド、そしてTmux+SQLiteで自作されたcafleetと、個別ツールが追従する周辺機能と、自作で補う実装パターンの両方が確認できる構成となった。

### 大規模コードベースにおける Claude Code の運用：ベストプラクティスと導入ガイド

https://claude.com/blog/how-claude-code-works-in-large-codebases-best-practices-and-where-to-start

Anthropicが公開したこのガイドは、数百万行規模のモノレポやレガシーシステムにおいてClaude Codeを効果的に運用するための戦略をまとめたものだ。最大の特徴は、従来のRAG（検索拡張生成）に依存せず、ライブのファイルシステムを直接ナビゲートする「エージェント的探索」を採用している点である。これにより、インデックスの遅れによる情報の古さを回避している。

成功の鍵として、5つの拡張ポイント（ハーネス）の構築が推奨されている。`CLAUDE.md`による階層化されたコンテキスト提供、フックによる学習の反映や自動チェック、Skillsによるタスクに応じた専門知識のオンデマンド読み込み、プラグインによる設定の組織内配布、MCPサーバーによる内部ツールやデータソースへの接続、の5つだ。さらにLSP（Language Server Protocol）の統合によるシンボルレベルの精度向上や、開発者体験（DevEx）チームによる専任の「エージェントマネージャー」の配置など、技術と組織の両面からのアプローチが詳述されている。Theme 1のハーネス設計論を、大規模コードベース運用という具体的文脈に落とし込んだ実践ガイドとして機能する。

### Claude Codeに「エージェント・ビュー」が登場：複数セッションの一括管理が可能に

https://claude.com/blog/agent-view-in-claude-code

公式ガイドが示す拡張ポイントは、UI側のAgent Viewにも反映されている。Anthropicは、CLIツール「Claude Code」に新機能「エージェント・ビュー（Agent view）」を追加した。この機能は、複数のエージェントを同時に動かす際、ターミナルのタブ管理や進捗把握が困難になるという課題を解決する。

ユーザーは`claude agents`コマンドまたは左矢印キーで一覧画面を呼び出し、どのエージェントが入力待ちか、あるいは実行中かを一目で確認できる。セッションをバックグラウンドに送る`/bg`コマンドや、一覧画面からセッションを完全に切り替えずに回答内容を確認しそのまま指示を返せる「Peek（覗き見）」機能が搭載された。これにより、プルリクエストの作成やダッシュボードの更新といった複数の並行タスクを、開発フローを中断することなく効率的に監督できる。本機能は現在、Pro、APIプラン以上のユーザー向けにリサーチプレビューとして提供されている。

### ファストモードによるレスポンスの高速化 - Claude Code ドキュメント

https://code.claude.com/docs/en/fast-mode

複数セッションを並列で動かすと、応答速度のボトルネックが現実問題になる。Claude Codeの「ファストモード」は、Claude Opusモデルの品質を維持したまま、レスポンス速度を2.5倍に向上させるAPI構成オプションだ。現在はリサーチプレビューとして提供されており、`/fast`コマンドで簡単に切り替えできる。

Opus 4.6（デフォルト）およびOpus 4.7（環境変数で有効化）に対応するが、コストは入力100万トークンあたり$30、出力$150と標準モードより高価に設定されている。利用はAnthropic Console APIおよびClaudeサブスクリプションプランの「追加利用（Extra usage）」枠に限られ、Amazon BedrockやGoogle Vertex AIなどのサードパーティ経由では利用できない。管理者は組織単位での有効化やセッションごとのオプトイン設定を強制でき、コスト管理と開発体験のバランスを調整できる。対話的なデバッグや迅速な反復作業など、速度が重要となる場面でのオプションとして位置づけられている。

### Visual Studio Code 1.120 リリース：Agents ウィンドウの Stable 到達と BYOK 機能の強化

https://code.visualstudio.com/updates/v1_120

### Claude Code の Agent Teams ぽいものを自分で作ってみたい、ついでに Codex でも使いたい。

https://zenn.dev/himkt/articles/00c6049542170b

ベンダー機能の追従と並行して、コミュニティは自作ツールで個別の運用課題を補っている。著者が公開した「cafleet」は、Claude Codeの実験的機能「Agent Teams」が抱える更新時の不安定さや、最新機能（1Mコンテキスト等）への追従の遅れといった課題を解決するために自ら開発したCLIツールだ。

cafleetはTmuxのペイン操作をラップし、複数のコーディングエージェント（Claude Code/Codex）をDirectorとTeammateという役割で協調させる。特筆すべきはパーミッション管理の手法で、Teammate側を「dontAsk」モードで起動し、Directorのセッションにコマンド実行の許諾を集約することで、複数ペインを行き来するストレスを排除している。メッセージング基盤にはSQLite3を採用し、1対1の送受信や一斉放送（broadcast）を実現。`uv tool install`で即座に導入可能であり、Codex側もカバーする柔軟性が特徴である。

5本を並べて読むと、マルチエージェント運用が「実験」から「日常運用」のフェーズに移行し、UI・速度・コスト管理が同時に整備されていることが見える。公式ベンダー（Anthropic/Microsoft）とコミュニティ（cafleet）が同じ問題に取り組んでいる事実は需要の強さを示し、「並列実行と監視」という共通テーマが個別機能の総和ではなく一貫した方向性として読める。ツールの足場が整ったことで、次の論点は「どう運用するか」——Theme 4の本番運用事例へと移っていく。

## 3. antirez DS4・nowokay・jolaが示すローカルLLMの「実用域」到達

ローカルLLMが「対話の成立」から「商用品質の代替」へと実用域を広げた週である。業界全体の現状認識、ハードウェア選定の指針、具体的な構築事例、フロントエンド級モデルのローカル稼働という4つの記事が、ハード・モデル・ツールの3面から到達点を確認できる構成として並んだ。

### 無料で使えるAI｢ローカルLLM｣の進化がすごいよって話

今週の35本を通じて見えてきたのは、AIコーディングの議論が「個別ツールの能力」から「ハーネス／Skills時代の設計原則と運用規律」へとさらに深く踏み込んだ事実である。Addy Osmaniの「Agent = Model + Harness」体系化、Google公式Skillsリポジトリの登場、クラシル・Codex・mrksyeの現場実装が同週に揃い、設計用語としてのハーネスは抽象論から具体実装の比較対象へと一段降りた。並行して、Agent View・Fast Mode・VS Code 1.120・公式運用ガイド・cafleetといった実行環境の整備が、複数エージェントの並列実行と監視を「実験」から「日常運用」のフェーズへ押し上げている。

本番運用の事例（LayerX・グロービス・モノタロウ・aircloset・LayerX VFS）が具体的な数値とともに重なり、ローカルLLM（DS4・Qwen 3.5・M4 Mac）が価格帯ごとに「何ができるか」の境界を明確化したことで、AI開発の足場は確実に厚みを増した。だが同時に、和田卓人氏の認知負債論、James Shoreの保守コスト論、Bryan Cantrillの「怠惰の美徳」喪失論、rust-langのLLMポリシーといった批判的論点が同じ強度で立ち上がり、生成量の増加が招く長期的負債が複数角度から検証された週でもある。生産性の数字と現場の感覚の落差、コードの短期速度と長期保守性のトレードオフは、引き続き議論の中心に居続けるだろう。

セキュリティ側では、Claude Mythos登場後の各国・各社・各セキュリティ研究者の同時並行的な動きが、単発の事件ではなく業界全体の地殻変動として可視化された。安野氏の政府批判、3メガバンクのアクセス権確保、NISTの定量評価、Kye Gomez氏のOpenMythos再現、Googleの実例報告、Metabaseの「露天掘り」現場、90日開示ポリシー機能不全の論考——これらは個別事件の集合ではなく、AI駆動セキュリティ時代の制度的・組織的な再設計が始まったことを示している。来週以降も、ハーネス設計の更新、本番運用事例の蓄積、認知・保守側の論点深化、そしてセキュリティ制度の再構成を、同じ視野で追っていきたい。
