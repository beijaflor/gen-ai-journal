## 1. Addy Osmani・Perplexity・技術評論社が示すAgent Skills設計原則

今週はAgent Skillsの設計論が一気に体系化された週となった。Addy OsmaniとPerplexityがそれぞれ独立に設計原則を公開し、Makefile形式の代替仕様、そして技術評論社からの公式入門書まで、設計・運用・記法・教育が同時並行で立ち上がった。

### AIエージェントにシニアエンジニアの規律を：Addy Osmani氏による「Agent Skills」の解説

https://addyosmani.com/blog/agent-skills/

GoogleのエンジニアAddy Osmani氏が、AIコーディングエージェントが陥りがちな「最短経路でのコード実装」という問題に対し、シニアエンジニアの規律をMarkdownで構造化する「Agent Skills」のフレームワーク論を公開した。骨格となるのは5つの設計原則である。

第一に「プロセス（工程）の重視」。仕様策定・テスト・レビューといった必須ステップを、エージェントが実行可能なチェックポイントとしてSkillに組み込む。第二に「反正当化テーブル」。エージェントが省略の言い訳として使う典型パターン（「時間がない」「テストは後で」など）を事前に列挙し、許容しないことを明示する。第三に「検証証拠の義務化」。タスク完了報告には実行ログやテスト結果といった一次証拠の添付を強制する。第四に「段階的開示」。すべてのコンテキストを最初に与えるのではなく、必要になった段階で詳細を読み込ませることでトークン消費を抑える。第五に「スコープの規律」。依頼されていないリファクタリングや構造変更を抑制し、変更影響範囲を意図された境界内に収める。

これらはOsmani氏がGoogle社内のエンジニアリング文化（Hyrum's Law、テストピラミッド、コードレビュー規律など）から抽出した規範を、Markdown形式でAIエージェントへ「コード化」したものとして位置づけられている。Claude CodeやCursorといった既存のエージェントツールに統合可能で、エージェントを「ジュニアレベルのコード生成器」から「シニアレベルの納品責任を持つ協働者」へ引き上げるための実践的な手引きである。読みどころは、原則の各項目が抽象論で終わらず、SKILL.md内に書き込める具体的な書式と禁止語彙のテーブルとして提示されている点だ。チームに導入する際、自分たちの開発文化のどの部分をSkillとして明文化するかを検討する出発点として機能する内容である。

### Perplexityにおけるエージェント・スキルの設計・改善・維持ガイド

https://research.perplexity.ai/articles/designing-refining-and-maintaining-agent-skills-at-perplexity

別の角度から、Perplexityは自社のエージェントシステム（Perplexity Computerなど）を支えるSkill群の構築・運用ガイドを公開した。Osmani氏の規律フレームに対し、こちらは本番運用から得られた現場知の体系である。

軸となるのは「Zen of Skills」。Zen of Pythonの裏返しのような独自原則で、「単純さより複雑さが機能となる」「明示的な命令より暗黙的なパターンマッチング（プログレッシブ・ディスクロージャ）を重視する」といった、通常のソフトウェア設計とは逆方向のヒューリスティクスが採用されている。構造面では、常時読み込まれる「インデックス（呼び出し条件）」、呼び出し時に読み込む「SKILL.md（本文）」、必要に応じて参照される「スクリプト/資産」という3階層を厳密に分け、トークンコストを最適化している。

特に強調されているのが「Load when...」から始まる呼び出し条件の設計と、評価セット（Evals）の重要性である。Skillの本文ではモデルが既に知っている一般知識を書かず、特定ドメインの「Gotchas（ハマりどころ）」と「独自の好み」だけを記述する。隣接ドメインとの混同を防ぐ負の例を含む評価セットを作成前に用意し、それに対する反復改善でSkillを磨き込むワークフローが、運用フェーズの中核として位置づけられている。

### SKILL.mk: Makefile形式を用いたAIエージェントスキルの仕様とツール

https://github.com/Teaonly/SKILL.mk

設計原則を踏まえると、記述フォーマット自体の選択も論点になる。SKILL.mkは、Markdownを規範とする現行のSkill定義に対し、Makefile構文で記述する代替仕様を提示するリポジトリである。

Makefileを採用する利点は、依存関係（DAG）の表現が記法レベルで備わっている点にある。エージェントが計画を立てる際、必要なRecipe（ターゲット）のみを動的に解決・読み込むことで、Markdown版で全文を毎回コンテキストに載せる方式と比べて実行時トークン消費を大幅に圧縮できる。実証例では最大85%の削減が報告されている。加えて、レシピ単位で実行統計を取得しやすく、Gitによる変更履歴管理も自然に行える。Skillの「Evolution Engineering（進化工学）」の基盤として、監査性と再現性を意識した提案である。

### Claude Codeで学ぶ Agent Skills入門

https://gihyo.jp/book/2026/978-4-297-15698-5

これらの設計論を日本語で体系的に学ぶための土台も整った。技術評論社から刊行された本書は、Agent Skillsを基礎から実践まで網羅した日本初の解説書である。

構成は、Agent Skillsが登場した背景とMCP（Model Context Protocol）との役割分担という前提整理から始まり、Claude Codeを用いたデザイン・ドキュメント生成（PDF・Word・Excel・PowerPoint）・システム開発の各領域でのハンズオンへと展開される。後半では公式の「skill-creator」ツールを使った独自Skillの開発・テスト・組織内配布までのライフサイクルを扱い、最後に「段階的開示」や「AIスロップ（AI生成のゴミ）」を避ける設計哲学へと踏み込んでいる。

## 2. DeepSeek V4・GitHub・Claude Code WebFetchで見える実質トークンコスト

トークン消費と価格構造、そしてエージェント内部の処理経路が、今週は具体的な数値で語られている。GitHubが自社運用で達成した62%削減、DeepSeek V4のキャッシュヒット時の極端な低価格、Claude CodeのWebFetchが裏でHaiku要約を挟んでいるという仕様暴露は、いずれも「実質トークンコスト」と「裏で何が動いているか」という共通指標で読み解ける。

### GitHub Agentic Workflowsにおけるトークン効率の改善手法

https://github.blog/ai-and-ml/github-copilot/improving-token-efficiency-in-github-agentic-workflows/

GitHubは、自社リポジトリのメンテナンスやCIで利用している「GitHub Agentic Workflows」のコスト最適化を、4つの工程に分解して報告している。最初のステップはAPIプロキシの導入である。すべてのLLMコールについて、入力・出力・キャッシュヒット・モデル種別を標準化されたフォーマットで記録・可視化することで、ワークフローごとのトークン消費を後段で比較可能にした。

次に取り組まれたのが、推論のたびに送信されるMCP（Model Context Protocol）ツール定義のスキーマ整理である。実際にエージェントが呼び出していないツール定義をプロンプトに同梱し続けるとコンテキストが膨張し、推論ごとに不要なトークン消費が発生する。GitHubは利用ログを基に未使用ツールを削除し、コンテキストを実需に合わせて絞り込んでいる。さらに、データ取得処理をLLMの推論ループから切り離し、決定論的なGitHub CLI（`gh`コマンド）に置き換えることで、ツールの判断に頼らずに必要な情報を確実に取得する設計へ移行した。

そして特筆すべきは「Effective Tokens (ET)」という独自メトリクスの定義である。単純なトークン数ではなく、モデルごとの単価・キャッシュヒット時の割引率・出力トークンの相対コスト比重を加味した「実質コスト指標」として設計されており、ワークフロー間の比較や最適化判断の共通言語として機能する。Issueの自動トリアージなど複数のワークフローで最大62%のコスト削減を達成したとされ、自社運用知見を外部開発者向けにも提示する形でメソドロジーが公開された格好である。読者にとって最も再現性が高い実装は、APIプロキシでの計測とMCPスキーマの棚卸しの2点で、これは個社の小規模ワークフローでも導入できる第一歩になる。

### Claude Opusの1786分の1!? DeepSeek V4のAPI価格がバグってるので全部計算してみた

https://qiita.com/lumichy/items/0252c195b4d0df797592

### あなたのClaude CodeのWebFetch、実はWebをちゃんと読んでいない

https://zenn.dev/zhizhiarv/articles/claude-code-webfetch-haiku-summary

価格と請求は数値で見えるが、エージェントの内部処理は数値化されにくい。Claude Code（v2.1.126時点）のWebFetchツールを詳細に分析した本記事は、ユーザー視点では見えにくい内部挙動を暴露している。

ユーザーが入力したURLは、メインモデル（Sonnet/Opus）に直接渡されるのではなく、まずClaude 3 Haikuによって要約・抽出処理が行われ、短縮された結果のみがメインモデルに到達する。この要約プロセスを回避できるのは「信頼されたドメイン（AWS、MDN等）」かつ「Markdown形式」かつ「10万文字未満」という極めて限定的な条件を満たすケースだけである。さらに、HTMLは先頭1MiB、Markdown変換後は先頭10万文字でサイレントに切り捨てられ、内部プロンプトには125文字の引用制限も組み込まれている。

GitHubのEffective Tokens、DeepSeekのキャッシュ価格、そしてWebFetchの内部要約パイプライン——3本に共通するのは「公称価格表や表示トークン数からは見えない実質コスト」が確実に存在する、という指摘である。請求書の額面ではなく、自分のワークフローで実際に何が処理されているかを計測する習慣が、エージェント運用の次のリテラシーとして問われている。

### 中国のオープンウェイトモデル「Kimi K2.6」がプログラミング競技でGPT-5.5やClaude、Geminiを破る

https://thinkpol.ca/2026/04/30/an-open-weights-chinese-model-just-beat-claude-gpt-5-5-and-gemini-in-a-programming-challenge/

「AI Coding Contest (AICC)」第12回大会で、中国Moonshot AIのオープンウェイトモデル「Kimi K2.6」が優勝した。競技種目はスライディングパズル形式で英単語を作る「Word Gem Puzzle」をリアルタイムで解くもので、特に大規模なグリッド（30×30）における動的なタイル操作能力が問われる課題である。

Kimi K2.6は、タイルを積極的に動かして単語を構築する「貪欲法」的なアプローチで最高得点を記録し、XiaomiのMiMo V2-Pro（2位）・GPT-5.5（3位）・Claude Opus 4.7（5位）といった主要フロンティアモデルを上回った。記事の分析によれば、欧米のフロンティアモデルは既存単語のスキャン能力には優れていたものの、タイルを動かして局面を打開する動的操作で差を付けられた。減点ルール（短い単語の作成）を正しく解釈できずに自滅したモデル（Muse Spark）もあり、「特定プロトコルや制約下での実行能力」がモデル間で顕著に分かれた。

このコンテストはコーディング能力の単一指標ではないが、オープンウェイト勢が商用フロンティアに対し、特定タスクでは明確に追いついている事実を示すマーカーとして読める。記事の論調も、性能差の縮小を慎重に確認する立場であり、「商用モデル一強」の前提を見直す必要がある時期に差し掛かっていることを暗示している。

### DeepSeek 4 Flash専用のMetal向け軽量推論エンジン「ds4」

https://github.com/antirez/ds4

モデル側の進歩と並んで、推論ランタイム自体を作り直す動きも出てきた。Redisの開発者として知られるSalvatore Sanfilippo (antirez) 氏が公開した「ds4」は、DeepSeek V4 Flashに特化したApple Silicon向けネイティブ推論エンジンである。

汎用GGUFランナー（llama.cppなど）とは設計思想が異なり、特定モデルに絞ることで深い最適化を図っている。MoE（混合専門家）部分のみを量子化して品質を保つ独自の2ビット量子化、SSDを活用して100万トークンのコンテキストを永続化・再開可能にする「Disk KV Cache」、OpenAI/Anthropic互換のサーバー機能などが特徴である。Apple Silicon（M3 Max/Ultra等）上で動作し、Claude CodeやPiといったコーディングエージェントからのローカル利用も想定されている。開発過程ではGPT-5.5の支援を活用しており、llama.cppの汎用主義とは別の道——「特定モデルに対して完結した体験を提供する」というアプローチ——を切り開いている。

### 次世代LLM「SubQ」発表：計算量がコンテキスト長に対し線形にスケールする初のサブクアドラティック・アーキテクチャ

https://subq.ai/introducing-subq

実装最適化の延長で、計算量そのものの前提を変えるアーキテクチャ提案も登場している。Subquadratic社が発表した「SubQ 1M-Preview」は、Transformerが宿命的に抱える「計算量がコンテキスト長の二乗に比例する」という制約を打破し、線形にスケールする初のフル・サブクアドラティック・アーキテクチャを採用している。

性能面では、1,200万トークン規模の研究段階で既存フロンティアモデルと比較してアテンション計算を約1,000倍削減、FlashAttention比でも52倍高速・計算量63%削減を実現する。RULER 128Kで95%精度を記録し長文読解・推論ベンチマークでClaude OpusやGPT-5.5に匹敵、SWE-Bench Verifiedでも81.8という高スコアを獲得している。コードベース全体を一度に読み込む「SubQ Code」と深層リサーチ用の「SubQ Search」がプライベートベータとして提供開始された。

性能（Kimi K2.6）・実用品質（Gemma 4）・実装最適化（ds4）・アーキテクチャ（SubQ）の4方向が同時並行で進んでいる事実が、今週のローカル実行フロンティアの輪郭を示している。SubQの線形スケーリングが本番品質に到達すれば、RAGやチャンキングといった従来回避策の前提自体が揺らぐ可能性もあり、長コンテキスト依存のワークフローを設計している開発者は注視すべき動きである。

## 5. GitHub Dominator・OWASP Juice Shop・SPECAが示すエージェント動作の証拠ベース検証

非決定的なエージェントが「成功した」と報告したとき、それを何で確かめるか。GitHubはドミネータ分析で自己評価82.2%に対し精度100%を達成し、Claude Code内蔵スキャンはOWASP Juice ShopでCritical級を80%以上検出、SPECAは仕様書からの監査でSherlockコンテストの新規バグ4件を発見した。検証の粒度を段階的に深めていく3本である。

### 決定論的でないAIエージェントの振る舞いを検証する：ドミネータ分析による「信頼レイヤー」の構築

https://github.blog/ai-and-ml/generative-ai/validating-agentic-behavior-when-correct-isnt-deterministic/

ソフトウェアテストの常識である「振る舞いの再現性」は、自律型AIエージェントの登場により通用しなくなっている。GitHub Copilot Coding AgentがUI操作（Computer Use）を伴う際、読み込み時間や操作パスの揺らぎによってタスクが成功してもスクリプトベースのテストが失敗する「偽陰性」が頻発する、という観察が起点である。

GitHubの研究チームは、コンパイラ理論の「ドミネータ分析」を応用した検証フレームワークを設計した。手順は3段階である。第一に、少数の成功トレースを収集する。第二に、それらをPrefix Tree Acceptor（PTA）というグラフ構造に統合し、各状態をLLMでセマンティック等価判定しながらマージする。第三に、タスク完了のために「必ず通過しなければならない状態（ドミネータ）」を、グラフ構造上から数学的に抽出する。エージェントの実行が、これらのドミネータ状態を所定順序で通過したかどうかを検証することで、表層的な手順の揺らぎを許容しながら本質的な達成を判定できる。

評価実験での結果は、エージェントによる自己評価（精度82.2%）を大きく上回る精度100%である。自己評価では「やったつもり」になっている逸脱が、ドミネータ分析では捕捉される。これは、エージェントを実験段階から本番環境のCI/CDパイプラインへ統合するための信頼レイヤー（Trust Layer）として機能しうる。読みどころは、コンパイラ理論の古典的概念がエージェント検証に転用されたという発想の方向性で、自社のエージェントワークフローで「成功とみなす条件」をどう構造化するかを考える際の参照点になる。検証データを集めるコスト、PTAマージのための等価判定コスト、ドミネータ抽出後の運用コストといった現実的な負担も明らかにされており、抽象論ではなく実装可能な手法として提示されている点が特徴的である。

### 「Claude Codeに全部やらせる時代が来た」のか検証してみた

https://zenn.dev/okazu_dm/articles/83957f1623ec4d

振る舞い検証の理論的枠組みに対し、実環境での検出率を実測した報告が出ている。セキュリティエンジニアの@okazu_dm氏は、Claude CodeのSlash Commandとして実装された「claude-security-scan」を、脆弱性診断練習用アプリOWASP Juice Shopで検証した。

結果は二面的である。脆弱性クラス単位では60〜70%の検出率、Critical級に絞ると80%以上を記録した一方、Juice Shopの全チャレンジ（約100件）ベースでは検出率は11%に留まった。SQLインジェクション・既知CVEを持つ依存パッケージ・ハードコードされたシークレットといった「明示的なパターンが残るタイプ」では強みを発揮するが、レースコンディション・DOM XSS・複雑なビジネスロジック由来の欠陥は構造的に検出が難しい。先行事例の「検出率100%」は対象の複雑性に依存するものと著者は分析している。

著者の結論は「安価なセキュリティ文化導入の入口としては有用」だが、「全てを任せられる段階ではない」というバランスの取れた評価で、LLMの揺らぎや網羅性の限界を理解した上での運用が前提である、という指針を提示している。

### SPECA: 仕様書からチェックリストを生成するエージェント型監査フレームワーク

https://github.com/NyxFoundation/speca/

実装レベルの検出に加え、仕様書を起点とした監査というアプローチもある。SPECAは、コードパターン依存の従来監査とは異なり、自然言語仕様書（RFCやEIPなど）からセキュリティプロパティを抽出し、それに基づいた「証明試行型（Proof-attempt）」の推論で実装の不備を特定するフレームワークである。

6つのフェーズ——仕様発見、グラフ抽出、プロパティ生成、コード位置特定、監査、レビュー——で構成される。SherlockのEthereum監査コンテストでの評価では、366人のプロ監査者が見逃した暗号学的脆弱性を含む4件の新規バグを発見し、全15件の既知脆弱性も特定している。RepoAuditベンチマークでもClaude 3.7/Sonnet 4.5を用いて従来最高精度に匹敵する88.9%を記録した。

3本に共通するのは「自己評価」「コードパターン」「人間レビュー」のどれか単独では検証として不十分であり、外部の数学的・実証的・仕様適合的な根拠が必要、という認識である。エージェント検証は「振る舞い」「実検出」「仕様適合」の三層で考える必要があり、自分のエージェント運用でどの層の検証が欠けているかを問い直す視点が、実務的な収穫となる。

## 6. Uncle Bob・Addy Osmani・Simon Willisonが問うAIコーディング委譲の認知コスト

AIエージェントへの委譲をどこで止めるかという議論が、複数の論者から独立に提示された週である。Addy Osmaniの「認知放棄」、Uncle Bobの役割転換論、Simon WillisonのVibe coding境界曖昧化への懸念は、それぞれ立場が異なるものの、「人間の理解と責任の所在」という共通の論点を扱っている。

### コグニティブ・サレンダー（認知の放棄）— AI時代のエンジニアが陥る「理解の負債」

https://addyosmani.com/blog/cognitive-surrender/

GoogleのAddy Osmani氏は、AIツール利用における「**認知のオフロード**（効率化のための委譲）」と「**認知のサレンダー**（批判的思考を伴わない放棄）」の決定的な違いを論じている。両者は表層的に似ているが、後者はAIの出力を自分の出力として無批判に受け入れ、独立した見解を形成しなくなる状態を指す。

氏はWharton SchoolやMITの研究を引用し、AIの存在が誤った自信を高め、脳の接続性や理解度を低下させる可能性を指摘する。これがエンジニアリング現場では、誰も内部構造を完全に把握していない「**理解の負債（Comprehension Debt）**」の蓄積に直結する。技術的負債とは異なり、理解の負債はコードベースが動いている間は表面化しないが、変更要求が来た瞬間に「誰も内部を説明できない」という形で噴出する。

Osmani氏が提案するのは、サレンダー化を防ぐための具体的なヒューリスティクス（指針）である。AIの回答を見る前に自分の期待値を設定する、AIのコードを新人の提出物として厳格にレビューする、AIに自身への反論を促す、といった主体性を維持する作法である。重要なのはAIというツールそのものではなく、それを利用する人間の「姿勢」である、というのが氏の中心的な主張だ。これらは個人レベルの規律として実装可能で、Theme 1のAgent Skills設計原則とも親和性が高い——「反正当化テーブル」が「サレンダーを構造で抑える」発想と通じている点も観察できる。

### アンクル・ボブ「プログラミングは終わった」：Vibe Coding時代の到来と開発者の行方

https://old.reddit.com/r/vibecoding/comments/1srfqm0/uncle_bob_its_over/

認知の放棄という枠組みは、職業としてのコーディングの再定義論にもつながる。「Clean Code」の提唱者Robert C. Martin (Uncle Bob) 氏が、AIの進化によって人間が直接コードを書く時代は終わったと宣言した動画がRedditで波紋を呼んでいる。Andrej Karpathy氏が提唱した「Vibe Coding」（コードそのものを意識せず雰囲気で開発を進めるスタイル）の潮流の中で、ボブ氏は「AIは人間の何倍も速くコードを書く。君たちがコードを書くことはもうない」と断言した。

ただし、本記事の焦点は挑発的な「終わった」宣言そのものではなく、それを起点としたコミュニティ議論にある。Reddit上の論争は概ね二つに割れている。一方は「開発者の役割は高レベル指示と生成物の監査にシフトした」と肯定する立場、もう一方は「AIの生成コードは非決定的で保守性が低く、依然として深い技術的知識による監督が必要だ」という懐疑的立場である。アセンブリから高水準言語への移行や、馬から自動車への転換（ジェボンズのパラドックス）を引き合いに、コード作成の低コスト化が逆にソフトウェア需要を爆発させ、新たな形の「英語による指示としてのプログラミング」が始まる、という分析もなされている。「終わった」というフックの裏で、議論は「監督役の質をどう保つか」に収斂しつつある。

### Vibe codingとAgentic engineeringの境界が曖昧になりつつある現状への懸念

https://simonwillison.net/2026/May/6/vibe-coding-and-agentic-engineering/

役割の議論に加え、実務家からは境界線そのものの消失が観察されている。Simon Willison氏は、「Vibe coding（コードを読まずに挙動のみで開発する）」と「Agentic engineering（専門家がAIを高度な道具として制御する）」の境界が急速に曖昧になっている、と指摘する。

かつては、専門家はAIが生成したコードを一行ずつ精査すべきだと考えられていたが、Claude Codeなどのエージェントの信頼性が向上するにつれ、専門家であってもコードを詳細にレビューせず、外部ライブラリや他チームのコンポーネントのように「ブラックボックス」として扱う傾向が強まっている、という観察である。Willison氏は、AIがテスト・ドキュメント・膨大なコミット履歴を瞬時に生成できるようになった今、成果物それ自体は「品質の証拠」とはならず、「誰かが継続的に使用し機能しているか」という実績が新しい信頼の源泉になりつつあるとも指摘する。

そして、実装コストの劇的な低下が設計プロセスのリスク許容度を高めるという楽観面と並んで、人間の「説明責任」の欠如や、監視を怠ることで将来的に致命的なミスを見逃す「逸脱の正常化」のリスクへの警鐘が鳴らされている。Osmani氏の「理解の負債」、Uncle Bobの「役割転換」、Willison氏の「境界消失」は、いずれも同じ現象を別の語彙で言語化したものであり、3者とも「ツールの問題ではなく姿勢の問題」という結論に独立に到達している。読者にとっては、自分が今どの種類の委譲をしているかを問い直す機会として読める。

## 7. Atlantic・GPU担保融資・「Claude Is Dead」が突きつけるAI業界の収益性と移行先選択

AI業界の収益・品質・財務構造の議論が、別々の対象を別々の指標で扱う形で同時期に並んだ。AtlanticはAnthropicの収益急増を根拠に「もうバブルではないかもしれない」と論じ、Tordable氏はClaude Codeの現場品質を痛烈に批判、別レポートはGPU担保融資の構造リスクと2026年の負債の壁を指摘している。3つの観察は対立しているのではなく、AI業界を多軸で評価するための材料として読める。

### AIは結局バブルではないのかもしれない ― 収益がハイプに追いつき始めた理由

https://www.theatlantic.com/economy/2026/05/ai-bubble-revenue-anthropic/687022/

The Atlanticの記事は、2026年時点の視点からAIバブル論の変遷を辿っている。かつてはドットコムバブルに例えられたAIセクターだが、Anthropicの「Claude Code」を筆頭とする自律型エージェントの登場により状況は変化した、というのが議論の出発点である。

具体的なシグナルとして、チャットボットから「実際にタスクを実行するエージェント」への進化が、ソフトウェア開発の生産性を実測可能なレベルで向上させていること、そしてAnthropicの収益成長率が歴史上のどの企業をも上回るペース（年換算300億ドル規模）に達していることが挙げられている。投資の重心は「期待」から「実益」へとシフトしており、コーディングから法律・マーケティングといった汎用的な知識労働領域へと用途が広がっているという観察である。

ただし記事は楽観論一色ではなく、依然として一部の専門家は不動産バブルのような過剰供給リスクを警告していると述べている。著者の論旨は「実務レベルでのAI導入スピードと成果が、否定派に立証責任を突きつける段階に到達している」という形で、「もうバブルではない」と断定するのではなく「バブル論の前提が変わった」という慎重な再評価である。年換算300億ドルという数値は印象的だが、その内訳がコーディング用途にどれだけ依存しているか、サブスクの解約率や法人契約の継続率がどう推移するかといった検証は今後の課題として残されている。

---

### 「クロードは死んだ」：AnthropicのClaude Codeが直面する性能劣化と開発者の不満

https://www.javiertordable.com/claude-is-dead/

収益面での楽観的見立てに対し、製品品質の現場では別の評価が出ている。元Google・MicrosoftのエンジニアでPauling.AIのCEOであるJavier Tordable氏は、「開発者の救世主」と目されたClaude Codeが「怠惰で使い物にならないゾンビ」と化したと論じる。

批判点は3つある。第一に、極端な利用制限。$20や$200の有料プランでも、数十分の作業でレート制限に達し実質利用不可になるという経験が、複数の開発者から報告されている。第二に、推論能力の低下（Dumbing-down）。AMDのAIディレクターによる大規模ログ分析を引き合いに、かつての洞察力が失われ、不要な書き換えや初歩的なミスが増加していると指摘する。第三に、経済的背景。VCによる「出血サービス」が終わり収益性を優先した結果、モデルの隠れたコンテキスト圧縮や思考プロセスの簡略化が行われている可能性を示唆する。Tordable氏は、開発者がGeminiやローカルLLMへ乗り換えを始めていると述べ、これをAIバブルにおける「神話の崩壊」を象徴する出来事と位置づけている。

### AIインフラに潜む金融バブル：GPU担保融資と2026年の「負債の壁」

https://financial-ai-bubble.pagey.site/The-Hidden-Financial-Bubble-in-AI-Infrastructure.pdf

収益と品質の議論とは別軸で、インフラ融資側の構造リスクも指摘されている。本レポートは、生成AIブームの裏側で進行している「インフラ投資の金融構造」に焦点を当てる。

論点は3つ。第一に、GPU担保融資の台頭。Coreweaveなどの新興クラウドプロバイダーが、NVIDIA製GPUを担保に数千億円規模の資金を調達し、さらなるハードウェア購入に充てる「再帰的なレバレッジ構造」を構築している実態が指摘される。第二に、2026年の「負債の壁」。多くのインフラ融資が2026年前後に返済期限を迎え、この時期までにAIアプリケーションが十分なROIを生み出せなければ、ハイパースケーラーの設備投資が減速し連鎖的なデフォルトや流動性危機（GPU Debt Wall）が発生するリスクがある、という予測である。第三に、データセンター不動産の過熱。テナント未確定での投機的なデータセンター建設が増加し、商業用不動産（CRE）市場での新たなリスク要因となっている点も挙げられている。

「収益増（Atlantic）」「品質劣化（Tordable）」「債務リスク（GPU Debt Wall）」は、対立する主張ではなく、別の指標による別の観察である。同じAnthropicが収益急増と現場での品質批判を同時に集めている事実は、収益と顧客体験の乖離を示唆する。読者にとっての含意は、自分の依存先（モデル・ベンダー）を、収益・品質・財務の3軸で同時に観察すべき、というものだ。GPU Debt Wallの2026年予測は確定的な未来ではないが、インフラ層のリスクが急に表面化したときに自分のスタックがどう影響を受けるかを想定しておく価値はある。

## おわりに

今週の24本を通じて見えてきたのは、AIコーディングの議論が「ツールの能力」から「運用と検証の規律」へとシフトしている事実である。Agent Skillsの設計原則が4方向から独立に体系化され、エージェント検証は自己評価ではなく外部根拠に依拠する方向で具体策が揃いつつあり、業務基盤側ではエージェント前提の機能が標準装備化のフェーズに入った。これらはいずれも「エージェントが動くこと」より「エージェントを安定して使い続けること」に重心が移っていることを示している。

並行して、Effective TokensやWebFetchの内部要約パイプラインのように、「公称価格や表示トークンからは見えない実質コスト」を計測する文化も育ちつつある。ローカル実行の選択肢——Kimi K2.6・Gemma 4・antirez ds4・SubQ——が同時に成熟したことで、商用フロンティアへの一強依存を見直す技術的余地も具体化した。Atlanticの収益論・GPU Debt Wallの債務論・Tordable氏の品質批判が同時に並んだ事実も、依存先を一指標で評価する時代の終わりを暗示している。

「コードを書くこと」自体への問い直しは、Osmani・Uncle Bob・Willisonがそれぞれ独立に提示した語彙——「認知のサレンダー」「役割転換」「境界消失」——として引き続き議論されている。技術の能力曲線がさらに上がる中で、人間側がどう構えるかという問いは、ツール選定や設計原則と同じ重みで日々の判断に影響していく。来週も、新機能のニュースと並行して、運用と検証の規律がどう更新されていくかを追っていきたい。
