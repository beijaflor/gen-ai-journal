GitHub Copilotを単なるコード補完ツールとしてではなく、プロジェクト固有のコンテキストを理解する「チームメンバー」として機能させるための高度な活用術がまとめられている。著者は、Copilotが認識する3種類の特別なファイル形式（`*.agent.md`、`*.instructions.md`、`*.prompt.md`）の使い分けを明確に定義し、開発者が直面しがちな「コンテキスト汚染」や「プロンプトの再入力」といった課題の解決策を提示している。

まず「Agent」は特定の専門知識やツール権限を持つカスタムAIの定義に使用される。フロントエンドやインフラなど領域別にAgentを分けることで、不要なコンテキストによる回答精度の低下を防ぎ、安全なツール利用（ファイル編集権限の制限など）を可能にする。「Instructions」はプロジェクト全体や特定のファイル（Rustの*.rsなど）に適用される永続的なルールを記述する。これにより「エラーハンドリングには特定のライブラリを使う」といった規約をAIに自動遵守させることができる。そして「Prompt」は定型的なタスク（単体テスト生成やリファクタリング）をテンプレート化し、変数入力や対話的なステップ実行を組み込むことで、作業品質の均一化と効率化を図るものである。

さらに、GitHubプルリクエストの自動レビューにおける`.github/copilot-instructions.md`の運用方法や、最近追加された「サブエージェント機能」の実験的検証についても触れている。サブエージェントに関しては、親エージェントが子の思考プロセスを直接見ることができないという現時点での制約を指摘しつつ、独立した検証（ダブルチェック）や専門的な調査に活用するパターンを提案している。

著者は、これらの機能を組み合わせて役割・ルール・タスクを構造化することが、AIとの協調開発における成果を最大化する鍵であると主張している。特にVS Codeのプロジェクト固有設定とグローバル設定を使い分ける視点は、複数のプロジェクトを抱えるWebエンジニアにとって即座に実用可能な知見である。
